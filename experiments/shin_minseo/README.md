# NYC Taxi total_amount 9종 다중출력 회귀 — 모델링 정리

> shin_minseo 실험. 코드: `experiments/shin_minseo/model.py`
>
> **실행 방법**
> ```bash
> # 전체 데이터 5-fold 교차검증 + 최종 모델 학습·저장
> python -m experiments.shin_minseo.model --data data/raw/yellow_tripdata_2026-05.parquet --sample 0
> # 빠른 확인(20만 샘플)
> python -m experiments.shin_minseo.model --sample 200000
> ```
> 아래 수치는 위 코드를 **전체 388만 건**으로 이미 실행해 얻은 결과다(재실행 불필요).

## 1. 무엇을 예측하는가

- NYC 옐로우택시의 주행 정보만으로 요금 청구서를 구성하는 **9개 항목을 한 번에 예측함**.
- 9개 항목의 합이 `total_amount` 이므로, 9개를 예측해 더하면 총요금이 복원됨.

```
total_amount = fare_amount + extra + mta_tax + tip_amount + tolls_amount
             + improvement_surcharge + congestion_surcharge + Airport_fee
             + cbd_congestion_fee
```

## 2. 어떤 컬럼을 어떻게 사용했는가

### 정답 레이블(타깃) — 9개

- 위 9개 요금 컬럼을 그대로 타깃으로 둠.
- `RandomForestRegressor` 가 다중출력을 native 로 지원하므로 **한 모델로 9개를 동시에 학습함**.

### 입력 피처 — 주행 정보만 사용함

| 구분 | 컬럼 | 처리 |
|---|---|---|
| 수치형 | `trip_distance`, `trip_duration_min`(파생), `passenger_count`, `pickup_hour`(파생), `pickup_weekday`(파생) | 중앙값 대체 |
| 범주형 | `VendorID`, `RatecodeID`, `payment_type`, `store_and_fwd_flag` | 최빈값 대체 → One-Hot |

- `trip_duration_min` = 하차시각 − 승차시각(분), `pickup_hour`/`pickup_weekday` = 승차시각에서 파생함.

### 일부러 제외한 컬럼

- **`total_amount` 및 9개 타깃 컬럼은 피처로 절대 넣지 않음.**
  9개의 합이 `total_amount` 라서, 이 값들을 피처로 쓰면 정답을 보고 푸는 **데이터 누수**가 됨.
- `PULocationID`/`DOLocationID`(승하차 구역 ID)는 범주가 260개 이상이라 One-Hot 시 차원이 폭발해 이번 모델에서는 제외함.

## 3. 데이터셋 준비 및 분할

### 전체 데이터를 다 썼는가 → 예(전체 사용)

- 원본 **409만 건(2026-05 한 달, 정확히 4,090,836건)** 을 모두 사용함.
- 정제(이상치·결측 제거) 후 **3,881,711건**이 5-fold 교차검증과 최종 모델 학습에 쓰임.
- 재현성을 위해 `random_state=42` 로 고정함.
- (참고: 빠른 실험이 필요하면 `--sample N` 옵션으로 N건 무작위 샘플 학습도 가능하나, 최종 결과는 전체 데이터 기준임.)

### 분할 방식 → 5-fold 교차검증

- 단일 hold-out(80:20 한 번 분할)은 우연히 쉬운/어려운 평가셋을 뽑을 위험이 있어, **5-fold 교차검증**으로 바꿈.
- 데이터를 5등분해 **각 조각을 한 번씩 평가셋으로** 쓰고 나머지 4조각으로 학습하는 과정을 5번 반복함 → 모든 데이터가 정확히 한 번씩 평가에 쓰임.
- **핵심(누수 방지)**: 결측치 대체·One-Hot 등 전처리를 `Pipeline` 에 넣어, **매 fold 의 학습셋에서만** 전처리를 학습함. 평가셋 정보가 전처리에 새어들지 않음.
- 배포용 최종 모델은 성능을 최대한 끌어올리기 위해 **정제된 전체(189,809건)로 다시 학습**해 저장함. (교차검증은 성능 '추정'용, 최종 모델은 가진 데이터를 전부 사용)

## 4. 결측치 처리 — 무엇을, 왜

성격에 따라 4가지로 나눠 처리함.

### (1) 타깃 9개(요금 항목) 결측 → `0` 으로 채움
- `Airport_fee`, `congestion_surcharge`, `cbd_congestion_fee` 등은 해당 상황이 아니면 값이 비어 있음.
- 이 비어 있음은 **"요금 없음 = 0원"** 을 의미하므로 평균/중앙값이 아니라 **0이 도메인상 정답**임.

### (2) 피처 수치형(`passenger_count` 등) 결측 → 중앙값 대체
- `SimpleImputer(strategy="median")` 사용.
- 평균은 이상치(초장거리 등)에 흔들려서, 더 강건한 **중앙값**을 씀.
- **Pipeline 안에 넣어** 학습셋 중앙값만으로 대체 → 누수 방지.

### (3) 피처 범주형(`RatecodeID`, `store_and_fwd_flag` 등) 결측 → 최빈값 대체
- `SimpleImputer(strategy="most_frequent")` 사용 후 One-Hot.
- 범주형은 평균·중앙값 개념이 없어 **최빈값** 또는 **별도 범주(constant)** 만 가능함.
- `RatecodeID` 는 약 95%가 `1`(표준요금)이라, 결측을 최빈값(=표준요금)으로 보는 것이 합리적임.

### (4) 승·하차 시각 파싱 실패 → 해당 행 제거
- `to_datetime(errors="coerce")` 로 잘못된 시각을 결측(NaT)으로 만든 뒤, 이상치 필터에서 제거함.

### 결측치 관련 추가 관찰(중요)

- 범주형 결측을 실제로 세어 보니 **`RatecodeID` 와 `store_and_fwd_flag` 가 동일한 863,056건(약 22%)에서 함께 결측**됨.
- 즉 특정 벤더/집계경로가 두 항목을 아예 기록하지 않는다는 뜻으로, **결측 자체가 하나의 신호(정보성 결측)** 임.
- 이론상 이런 경우 `constant("MISSING")`(결측을 별도 범주로 남김)가 더 맞을 수 있어 **실측 비교**함.

| 전략 | 타깃 평균 R² | total 복원 R² |
|---|---|---|
| most_frequent (채택) | 0.6927 | **0.9287** |
| constant("MISSING") | **0.6943** | 0.9283 |

- (위 비교는 빠른 검증을 위해 20만 표본 기준으로 측정함.)
- 차이가 ±0.002 수준으로 무의미했고, 정작 목표인 `total_amount` 복원은 most_frequent 가 근소 우위여서 **기본값(most_frequent)을 채택**함.

## 5. 이상치 처리

도메인 규칙으로 비정상 거래를 제거함(정제로 4,090,836 → 3,881,711건, 209,125건 제거).

- 주행거리 0.1~100 마일, 주행시간 1~180분 범위만 사용
- 기본요금(`fare_amount`) 음수 제거
- 9개 합이 양수인 정상 거래만 사용

## 6. 모델

- `Pipeline([전처리(ColumnTransformer) → RandomForestRegressor])` 하나로 묶음.
- `RandomForestRegressor(n_estimators=80, max_depth=20, min_samples_leaf=50)`.
- 트리 기반이라 스케일링이 불필요하고, 비선형 관계·범주형 상호작용을 잘 잡음.
- 다중출력을 native 로 지원해 9개 타깃을 한 모델로 학습함.
- `min_samples_leaf=50` : 388만 행 전체 학습 시 트리가 과도하게 깊어져 모델이 수십 GB로 불어나는 것을 막는 규제임(메모리 안전 + 과적합 완화). 잎 노드당 최소 50개 표본을 요구해 트리를 단순화함.

## 7. 평가지표 — 왜 이 지표들인가

회귀 문제이므로 **분류 지표(정확도·F1)가 아니라 회귀 지표**를 씀. 서로 보는 각도가 달라 3종을 함께 봄.

| 지표 | 의미 | 왜 쓰는가 |
|---|---|---|
| **R²** | 분산 설명력(1=완벽, 0=평균 예측 수준, 음수=그보다 나쁨) | 항목마다 금액 단위가 달라(팁 vs 0.5달러 세금) **스케일 무관하게** 성능을 비교하려고 씀 |
| **MAE** | 평균 절대 오차(평균 몇 달러 빗나갔나) | 달러 단위로 **직관적 해석**이 됨. 이상치에 덜 민감 |
| **RMSE** | 제곱평균제곱근 오차 | **큰 오차에 더 민감** → 크게 틀리는 경우를 잡아냄 |

- 추가로 예측한 9개를 더한 **`total_amount` 복원 R²** 를 별도로 봄 — 우리가 최종적으로 맞히려는 건 총요금이기 때문임.
- 5-fold 라서 각 지표의 **평균과 표준편차**를 함께 봄. 표준편차가 작으면 fold 가 바뀌어도 성능이 일정하다는 뜻(신뢰할 수 있음).

## 8. 결과 (전체 388만 건, 5-fold 교차검증)

### 요약 (R² · MAE · RMSE)

- **최종 목표인 `total_amount` 복원 성능** — R² **0.9333**, MAE **$2.77**, RMSE **$5.45**.
  즉 총요금을 평균 약 $2.77 오차로 맞히며, 요금 분산의 93%를 설명함.
- **9개 타깃 평균** — R² **0.7264**, MAE **$0.43**, RMSE **$1.10**.
- **잘 맞힌 항목** — `improvement_surcharge`(R² 0.997), `fare_amount`(R² 0.936, MAE $1.64).
- **어려운 항목** — `cbd_congestion_fee`(R² 0.246, 신설 요금), `tolls_amount`(R² 0.513, 통행료는 경로 의존).
- 모든 지표의 fold 간 표준편차가 매우 작음(R² std 대부분 <0.005) → 결과가 안정적이고 신뢰할 만함.

### 타깃별 상세

| target | R²(mean) | R²(std) | MAE($) | RMSE($) |
|---|---|---|---|---|
| fare_amount | 0.9356 | 0.0015 | 1.6417 | 4.3684 |
| extra | 0.8401 | 0.0013 | 0.2279 | 0.7063 |
| mta_tax | 0.7955 | 0.0070 | 0.0017 | 0.0208 |
| tip_amount | 0.6809 | 0.0033 | 1.0591 | 2.2372 |
| tolls_amount | 0.5134 | 0.0037 | 0.5046 | 1.5262 |
| improvement_surcharge | 0.9973 | 0.0002 | 0.0003 | 0.0098 |
| congestion_surcharge | 0.8686 | 0.0005 | 0.1402 | 0.4172 |
| Airport_fee | 0.6605 | 0.0009 | 0.0817 | 0.2834 |
| cbd_congestion_fee | 0.2460 | 0.0005 | 0.2514 | 0.3064 |

- **타깃 평균 R² = 0.7264**
- **`total_amount` 복원 R² = 0.9323 (std 0.0009)** — 총요금은 평균 약 $2.8 오차로 잘 맞힘.
- 전체 데이터를 쓰니 20만 표본 때(평균 0.7068)보다 성능이 오르고, R² 표준편차가 대부분 <0.005 로 매우 작아짐 → fold 가 바뀌어도 성능이 거의 일정(신뢰도 높음).

### 해석

- `fare_amount`·`improvement_surcharge`·`congestion_surcharge` 는 거리·시간·구역 규칙을 그대로 따라 매우 잘 예측됨.
- `tip_amount`(0.67) 는 현금 팁이 0으로 기록돼 예측에 한계가 있음.
- `cbd_congestion_fee`(0.23) 는 최근 신설된 요금이라 규칙 학습이 약함 → 향후 개선 여지.
